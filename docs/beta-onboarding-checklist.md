# ScopeHouse Beta Onboarding Checklist v1

## Purpose

Create a simple onboarding checklist for early beta users.

This checklist should help one person onboard users in a repeatable way.

It should work for:

- homeowner beta users
- solo pros
- design partners
- founder-led service users who may convert into product users

## Core rule

Do not make onboarding broad.

The goal is to get the user to the first clear success moment fast.

## First success moment

The first session is successful if the user leaves with:

- a live project created in ScopeHouse
- intake captured clearly enough to work from
- a visible scope draft or structured project record started
- a clear next action agreed

## Pre-onboarding qualification

Before onboarding, confirm:

- the lead passed qualification
- the project is active or starting soon
- the user has enough authority to participate meaningfully
- the likely route is clear, beta, design partner, or service-led setup
- the next session is worth the founder’s time

## Required data before the session

Collect or request:

- project type
- project stage
- rooms or phases involved
- budget range if known
- timing constraints if known
- current quotes if available
- current notes, docs, or spreadsheets if available
- main pain point

## Common onboarding structure

### Step 1, confirm fit and context

At the start of the session, confirm:

- what project is being discussed
- what stage it is in
- what part feels least under control
- what the user wants from the session

### Step 2, create the project

Do:

- create project record
- name the project clearly
- set basic project stage and type

### Step 3, run intake

Capture:

- goals
- rooms or phases
- priorities
- constraints
- budget range
- timing frame

### Step 4, get to structure fast

Do:

- move from rough notes into a clearer project record
- identify the main control gap
- create a scope draft or equivalent working structure

### Step 5, confirm the first win

Show the user:

- the project now has one place to live
- the record is cleaner than before
- the next session or next action is clear

### Step 6, set follow-up

Before ending, confirm:

- next action
- who owns it
- next session date if needed
- what material the user needs to send before the next touchpoint

## Segment checklist

### Homeowner beta user

Best when:

- the project is meaningful in cost or complexity
- quote comparison or decision confusion is active

Checklist:

- confirm the project is a real renovation, not a vague future idea
- collect rooms, priorities, constraints, and budget range
- ask whether quotes are already in play
- identify the most confusing current decision
- create project record
- run intake
- create first scope structure
- agree on one follow-up action

First success moment:

The homeowner sees their renovation moved out of scattered notes and into one structured project record.

### Solo pro beta user

Best when:

- they want cleaner project control across client work
- they do not want heavy contractor software

Checklist:

- confirm their current workflow and current tools
- identify where records break down across client projects
- choose one real project to use for onboarding
- create project record
- run intake or project setup
- show decision, quote, or change tracking path
- agree on the client-facing value they want most
- set next session or next test step

First success moment:

The pro sees one real client project become easier to control.

### Design partner onboarding

Checklist:

- confirm design partner expectations
- confirm openness to follow-up and feedback
- note what kind of feedback is most useful
- create project record
- run the first workflow session
- capture friction and reactions live
- set a feedback checkpoint within 7 days

First success moment:

The design partner gets a working project started and understands that their feedback will shape the product.

### Founder-led service user converting into product user

Checklist:

- confirm the service scope
- gather project artifacts before the session if possible
- set up the project in ScopeHouse with the user
- walk through the cleaned-up record
- show what they can continue to use after the service work
- define whether they should stay in product, service, or both

First success moment:

The user sees a cleaner project record and understands how ScopeHouse continues after the service work.

## Support expectations

Tell the user:

- this is an early product
- the goal is to get the core workflow working with them, not leave them alone in setup
- follow-up is part of the process for early users
- feedback is expected and useful

## Follow-up checkpoints

### Within 24 hours

- send summary of what was done
- send next action
- send any requested follow-up items

### Within 7 days

- confirm whether the user returned
- confirm whether the next action happened
- capture friction or blockers

### Within 14 days

- decide whether the user is active, stuck, parked, or ready for deeper onboarding

## Post-session logging

After every onboarding session, log:

- segment
- project stage
- top pain point
- what was completed
- whether the first success moment happened
- next action
- next session date if any
- risk of drop-off
- product friction found

## Drop-off risks

Watch for:

- unclear project timing
- too much setup effort in first session
- weak project pain
- missing user authority
- no next action agreed
- user leaves without a visible win

## Operating reminder

A good onboarding session should feel like progress, not homework.

The user should leave thinking:

- my project is clearer now
- I know what happens next
- this is helping me get control
